# Core Beliefs — SalesHarness

> 이 문서의 원칙은 **모든 설계와 구현 결정에 우선**합니다.
> 트레이드오프 상황에서 이 원칙을 위반하는 선택은 하지 않습니다.
> AI가 코드를 생성할 때 반드시 이 원칙을 준수해야 합니다.

---

## Belief 1: 고객 데이터는 성역이다

> "편의를 위해 보안을 타협하지 않는다."

금융 고객사의 데이터는 비즈니스의 핵심 자산이다.

### 구체적 규칙
- 고객사 데이터는 **서버사이드에서만** 처리. 클라이언트에 민감 데이터 노출 금지.
- 모든 DB 테이블에 **RLS (Row Level Security)** 적용. 예외 없음.
- API Key, 토큰 등 시크릿은 **환경변수**로만 관리. 코드에 하드코딩 절대 금지.
- 고객 데이터 접근은 **감사 로그(Audit Log)**로 기록.
- AI에 전달되는 프롬프트에 **고객 개인정보(주민번호, 계좌번호 등)**를 포함하지 않음.
- 에러 메시지에 내부 구조나 데이터를 노출하지 않음.

### 위반 시
→ 즉시 수정. 다른 기능 개발보다 보안 이슈 수정이 우선.

## Belief 2: AI는 보조자, 사람이 결정자

> "AI가 만든 것은 초안이다. 최종본은 사람이 만든다."

AI는 영업직원의 업무를 **대체**하는 것이 아니라 **증폭**시킨다.

### 구체적 규칙
- AI 생성 콘텐츠는 항상 **편집 가능한 상태**로 제공. 읽기 전용 최종본 없음.
- 모든 AI 출력에 **"AI 생성" 표시**를 붙임.
- AI가 **자동으로 이메일을 보내거나, 계약을 체결하거나, 데이터를 삭제**하지 않음.
- 사용자의 **승인/수정/거절** 피드백을 저장하여 품질 개선에 활용.
- AI 추천은 **근거(이유)**와 함께 제시. "이 상품을 추천합니다"가 아니라 "A사의 IT 현대화 계획에 맞춰 이 상품을 추천합니다".

### 위반 시
→ AI 자동 실행 기능 추가 요청이 와도 거절. 사용자 확인 단계 필수.

## Belief 3: 단순함이 이긴다

> "기능을 하나 더 넣는 것보다, 기존 기능을 더 쉽게 만들어라."

영업직원은 IT 전문가가 아니다. 도구가 복잡하면 안 쓴다.

### 구체적 규칙
- **3클릭 룰**: 핵심 기능은 3번의 클릭/입력 이내에 도달.
- 한 화면에 **하나의 핵심 태스크**. 여러 기능을 한 페이지에 욱여넣지 않음.
- 설정/옵션은 **최소화**. 스마트 기본값(Smart Defaults)을 우선.
- 에러 메시지는 **행동 지향적**. "Error 500"이 아니라 "다시 시도해주세요. 문제가 지속되면 관리자에게 문의하세요."
- 빈 상태(Empty State)에는 **다음 행동 가이드** 제공. "제안서가 없습니다" → "첫 제안서를 만들어볼까요?"
- 로딩 상태에는 **진행률 또는 예상 시간** 표시 (특히 AI 생성 시).

### 위반 시
→ "고급 사용자용" 기능이라도 기본 화면을 복잡하게 만들면 안 됨. Progressive Disclosure 적용.

## Belief 4: 한국어가 기본이다

> "번역체가 아닌, 자연스러운 한국어."

사용자는 한국의 금융 영업직원이다. 모든 인터페이스와 콘텐츠는 한국어가 기본.

### 구체적 규칙
- UI 라벨, 버튼, 메뉴는 **한국어**. 불필요한 영어 사용 금지.
  - ❌ "Dashboard" → ✅ "대시보드" 또는 "홈"
  - ❌ "Submit" → ✅ "저장" 또는 "생성"
  - ❌ "Pipeline" → ✅ "영업 현황" (단, 업계에서 통용되는 용어는 예외)
- AI 생성 콘텐츠는 **한국 비즈니스 문화**에 맞는 톤. 존댓말 기본.
- 날짜 형식: `YYYY년 MM월 DD일`, 통화: `원(₩)`
- **코드의 변수명/함수명은 영어**. 주석은 한국어 허용.
- 금융IT 업계 전문 용어는 **한국 시장 표준**을 따름.

### 위반 시
→ 영어 UI 라벨 발견 시 한국어로 교체. 번역체 문장은 자연스러운 한국어로 수정.

## Belief 5: 코드 품질은 균형이다

> "MVP는 속도, 이후는 품질. 하지만 기반은 처음부터 단단하게."

### 구체적 규칙

**처음부터 지키는 것 (Non-negotiable):**
- TypeScript strict mode — 타입 안전성
- 환경변수로 설정 관리 — 하드코딩 금지
- 에러 핸들링 — try/catch, 사용자 친화적 에러 메시지
- 컴포넌트 분리 — 한 파일 800줄 이하
- 불변성(Immutability) — 객체 직접 변경 금지

**점진적으로 강화하는 것:**
- 테스트 커버리지: Phase 0-1은 핵심 로직만 → Phase 2부터 80% 목표
- 코드 리뷰: Phase 0은 셀프 리뷰 → Phase 1부터 체계적 리뷰
- 성능 최적화: Phase 0-1은 기능 우선 → Phase 2부터 최적화
- 문서화: 코드 주석은 최소, README는 Phase별 업데이트

---

## 원칙 간 우선순위

충돌하는 상황에서의 판단 순서:

```
1. 보안 (Belief 1)          ← 절대 우선
2. 사람이 결정 (Belief 2)    ← AI 자율 실행 금지
3. 단순함 (Belief 3)        ← 복잡한 기능보다 쉬운 UX
4. 한국어 (Belief 4)        ← 자연스러운 한국어
5. 코드 품질 (Belief 5)     ← 상황에 맞는 균형
```

예시:
- "보안 절차가 UX를 복잡하게 만든다" → **보안 우선**, UX는 보안 내에서 최적화
- "AI가 자동으로 보내면 더 편하다" → **사람 확인 필수**, 편의보다 안전
- "영어가 더 짧고 깔끔하다" → **한국어 우선**, 한국어로 표현
